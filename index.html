THE DIFFERENCE

Agents vs
Automation.

SCRIPT

"Send this email."

Does exactly one thing. Nothing more.

AGENT

"Make sure this customer gets a response within 4 hours."

Figures out the how — checks if someone responded, drafts a reply, escalates if complex.

Goal-driven, not instruction-driven.

04 / 13
EVERY AGENT NEEDS

Three
Components.

Perception.
APIs, databases, documents — how the agent sees the world. Only as good as what it can observe.
Decision Logic.
Structured rules for routine cases. Invoke the LLM only for ambiguous situations.
Action Interface.
How the agent affects the world. Every action: logged, reversible, permissioned.
05 / 13
ACTION LAYER

Tools.

  • Tools = functions the agent can call
  • Each tool does one thing: send email, query DB, create event
  • The model doesn't execute tools — it requests them
  • Good design: one job, clear inputs/outputs, no silent failures
// The model outputs:

THE FLOW

How Tool Calling
Works.

Agent Request
→
Validate
→
Execute
→
Return Result
  • Orchestration layer checks permissions & validates parameters
  • Executes the function, captures the result
  • Feeds structured data back — agent reasons about what to do next
  • The agent knows what the tool does, not how

PERSISTENCE

Context &
Memory.

CONTEXT WINDOW

Active work — the current task, recent messages, immediate state.

Limited. Even 200K tokens fills up.

EXTERNAL MEMORY

History — completed tasks, logs, summaries, past decisions.

Doubles as an audit trail for debugging.

09 / 13
CHOOSING RIGHT

Model
Selection.

FRONTIER
Opus · GPT-4o
Complex reasoning, planning, ambiguous decisions. Slow, expensive.
MID-TIER
Sonnet · 4o-mini
Most agent tasks. Good balance of capability, speed, and cost.
SMALL / FAST
Haiku · Flash
Classification, routing, simple extraction. High-volume, cheap.

Not every task needs the most powerful model.

EDGE CASES

Like a company — you don't send the CEO to answer every support ticket.

  • A 10x cheaper model that handles 80% of cases saves you a ton
  • Latency matters for user-facing agents — faster model = better UX
  • Consider: cost · latency · complexity · tool calling support

MOST IMPORTANT STEP

Plan Before
You Execute.

Agent Plans
→
Human Reviews
→
Agent Executes

Most agent failures happen because people skip planning.

Planning is cheap. Recovering from bad execution is expensive.

12 / 13
DESIGN FOR IT

Failures &
Guardrails.

Retry.
Transient failures — network timeouts, rate limits. Exponential backoff.
Escalate.
Low confidence? Stop and ask a human. Human-in-the-loop.
Fail Safe.
Never destroy data. Always fail in a way that's recoverable.

Guardrails = hard limits the agent can't bypass. Permissions = role-based access. Make failures observable.
